# 工作区（Workspace）与智能体模式（Agent Mode）设计构想

> 状态：构想 / 设计草案（未实现）
> 目标读者：Aetherlink 开发者
> 结论先行：**先做工作区（IDE 式），它是智能体模式的硬前置。智能体与主聊天应隔离，二者共享"能力层"，隔离"领域层 / 执行层 / 存储 / 任务态 UI"。**

---

## 0. 背景与目标

Aetherlink 目前是一个移动端聊天 App。未来希望引入：

1. **工作区（Workspace）**：类似 IDE 工作区——一个根目录 + 文件树 + 文件查看/编辑 +（可能的）终端，所有操作围绕这个目录展开。
2. **智能体模式（Agent Mode）**：类似 Devin 云端 / Codex 那种能自主多步执行、读写文件、跑命令、产出结果（diff / PR / 产物）的模式。

两者关系：**工作区是智能体的"手术台"。** 智能体的本质是「读文件 → 改文件 → 跑命令 → 看结果」，没有工作区就无处下手。因此实现顺序应为：**先工作区，后智能体。**

---

## 1. 核心原则

### 1.1 智能体必须与主聊天隔离

聊天和智能体是两种本质不同的东西，强行塞进同一套数据 / UI / 状态会在后期造成巨大维护成本。

| 维度 | 聊天模式 | 智能体模式 |
|---|---|---|
| 生命周期 | 一问一答、用户在场 | 长时运行、自主多步、用户离场也在跑 |
| 数据模型 | message 列表 | 执行树：plan → steps → tool calls → observations → 子任务 |
| 状态 | 基本无（流式中/完成） | 完整状态机：running / paused / blocked / failed / done，可中断/恢复/回溯 |
| 副作用 | 无 | 改文件、跑命令、调外部 API（真实副作用、可能花钱、可能出错）→ 需沙箱 / 权限边界 / 审计 / 人工审批(HITL) |
| 并发 | 基本串行 | 多任务并行，每个独立上下文 / 工作区 |

**隔离边界：**

- **共享「能力层」**：LLM provider 调用、工具注册表（tool registry）、鉴权 / 密钥、设计系统组件（如 `lib/shared/widgets/`）、**工作区后端（见第 2 节）**。
- **隔离「领域层 + 执行层 + 存储 + 任务态 UI」**：
  - 领域层彻底分开：`features/agent/`（新） vs `features/chat/`（现有），各自 model / repository / controller，不共用实体。
    - chat：`Conversation / Message`
    - agent：`AgentTask / AgentRun / AgentStep / ToolCall / Artifact`，状态机驱动
  - 存储分表/分库：agent 执行轨迹数据量大、写得频繁，单独存。
  - 执行隔离：每个 agent run 独立工作区 + 独立上下文窗口。
  - UI 隔离但入口可融合：任务列表 / 任务详情独立；允许「在聊天里召唤一个 agent 任务」作为入口，但任务本身跳到 agent 那套界面运行。

> 类比 Devin / Codex：聊天是轻量入口，真正的活儿在独立的 session/task 实体里，有自己的状态机、工作区、事件流、产物面板和审批。聊天可以触发它、查看它，但不是同一个东西。

**一句话结论**：聊天 ⊂ 产品，智能体 ⊂ 产品，二者是**兄弟模块，不是父子**。

## 2. 工作区架构（重点）

### 2.1 设计目标

支持两类形态：

1. **纯手机工作区**：文件在手机本地。
2. **联动工作区**：手机 ↔ 桌面端 或 手机 ↔ Termux，把「跑命令」能力引入。

核心设计：**抽象一个统一的「工作区后端」接口 `WorkspaceBackend`，底下挂多种实现。** 上层（文件树 UI、编辑器、diff、聊天 @文件、以及未来的 agent）只依赖该接口，完全不关心底层是本地还是远端。

### 2.2 统一抽象接口

```dart
abstract class WorkspaceBackend {
  // 文件
  Future<List<FileEntry>> listDir(String path);
  Future<String> readFile(String path);
  Future<void> writeFile(String path, String content);
  Stream<FileChangeEvent> watch(String path); // 可选

  // 执行（纯本地工作区可不支持，抛 UnsupportedError）
  Stream<ExecOutput> exec(String cmd, {String? cwd});

  // 能力声明，UI 据此显示/隐藏终端等功能
  WorkspaceCapabilities get capabilities; // { canExec, canWatch, isRemote }
}
```

上层只面向 `WorkspaceBackend` 编程；三种形态 = 三个实现类，UI / agent 代码零修改即可切换。

### 2.3 三种后端实现

#### ① 纯手机工作区（本地）

- 后端：
  - Android → **SAF（Storage Access Framework）**：用户用系统文件选择器（`ACTION_OPEN_DOCUMENT_TREE`）授权一个文件夹，拿到目录树 URI（`content://...`）+ 持久化权限，之后在这棵树上读写/建文件。
  - iOS → `UIDocumentPicker` + security-scoped bookmark（与 SAF 不同的一套 API）。
- `canExec = false`（手机本地跑不了命令）。
- 用途：轻量看/改文件、笔记、配置、聊天 @文件进上下文。
- 注意坑：
  - SAF 基于 URI / `DocumentFile`，不是普通路径，API 啰嗦且遍历大目录**慢**，文件多的代码项目体验打折。
  - iOS / Android 两套 API，跨端要各写一份。

#### ② 手机 ↔ Termux 联动（同机，性价比最高的"能跑命令的工作区"）

Termux 是装在同一台安卓手机上的 Linux 环境。

- 联动方式：
  - (a) **共享目录**：Termux 把项目放共享存储，App 用 SAF 读写同一目录 → 文件层打通。
  - (b) **Termux 里跑本地 server**：Termux 起一个 `sshd` 或小型 HTTP/WebSocket 守护进程，监听 `127.0.0.1:<port>`；App 通过 localhost 连它来 `exec` 命令、读输出。
  - 最干净组合：文件走共享目录（或同一 server 的文件 API），命令走 localhost server。
- `canExec = true`。
- 用途：手机上就能 build / git / 跑脚本 → 让 agent 在纯手机上也能干真活。

#### ③ 手机 ↔ 桌面端联动（跨设备，最接近 Devin 体验）

- 桌面端跑一个 agent/daemon（自研伴随程序，或复用 SSH），手机 App 通过网络（局域网 / 内网穿透 / 中继）连接。
- 后端实现：对接桌面 daemon 的 file API + exec API（本质同 ②(b)，只是地址从 localhost 换成桌面/远端地址）。
- `canExec = true, isRemote = true`。
- 用途：重活在桌面跑（算力 / 真实开发环境），手机当遥控器和查看器。

### 2.4 为什么这套设计成立

- **②③ 在协议层几乎一样**（都是「远端有个 daemon，提供 file API + exec API」），区别只是连 localhost 还是连远端地址 → **远端协议做一次，Termux 和桌面端都能用**。
- **①** 是「纯文件、无 exec」的特例。
- agent 模式面向 `WorkspaceBackend` 编程，在哪跑由用户选的后端决定，**agent 代码零修改**。

## 3. 落地顺序（建议）

### 阶段 A：工作区（先做）

1. 定 `WorkspaceBackend` 接口 + 文件树 / 编辑器 UI；先接 **① 本地 SAF**（最快出东西，先只读也行）。
2. 把**聊天接上工作区**：`@文件` 进上下文、引用代码行 → 立刻让聊天变强，同时验证抽象是否好用。
3. 设计**远端 daemon 协议**（file API + exec-over-WebSocket）。
4. 先用 **② Termux** 验证（同机、无网络复杂度、最容易调通）。
5. 把同一协议指向 **③ 桌面端 daemon**（加配对/发现/安全）。

### 阶段 B：智能体（后做）

6. 建 `features/agent/` 领域层：`AgentTask / AgentRun / AgentStep / ToolCall / Artifact` + 状态机。
7. agent 执行层接到 `WorkspaceBackend` 上（在用户选定的后端里读写/跑命令）。
8. 任务态 UI：任务列表 + 任务详情（plan / 日志 / diff / 产物 / 审批点）。
9. 聊天侧加「召唤 agent 任务」入口。

---

## 4. 需要尽早决策的风险点

- **安全 / 鉴权**：`exec` 接口 = 远程执行命令，必须 token / 配对，绝不能裸奔（尤其桌面端走网络）。
- **连接发现 / 配对**：② localhost 简单；③ 需解决配对（扫码 / 局域网发现 / 内网穿透 / 中继）。
- **iOS 限制**：Termux 是安卓独有；iOS 只能走 ③ 桌面联动或云端；本地工作区两端 API 也不同。
- **SAF 性能**：大目录遍历慢，需考虑缓存 / 懒加载 / 索引。
- **副作用治理（agent）**：沙箱、权限边界、审计日志、人工审批（HITL）、可中断 / 可回滚。

---

## 5. 一句话总结

> 工作区不是"聊天的工作区"，是"产品的工作区"，是智能体的硬前置。
> 用一个 `WorkspaceBackend` 抽象统一「本地 SAF / Termux / 桌面端」三种后端；
> 共享能力层（模型、工具、组件、工作区），隔离领域层 / 执行层 / 存储 / 任务态 UI；
> 先把工作区做出来并接进聊天，未来再把智能体挂到同一个工作区上。
